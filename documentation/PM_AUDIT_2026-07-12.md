# ChemVault Model PM Audit

Date: 2026-07-12

## Audit scope

This review applies the PM Skills intended-versus-implemented and launch pre-mortem methods to the current website, Windows desktop app, Apple native app, quantum workflow, release pipeline, product telemetry, and shipping documentation.

Evidence checked:

- Current `main` and `test` branch alignment.
- Current successful CI, security, Apple native, and Windows packaging workflows.
- Local regression suite and dependency audit.
- Production website version manifest and a live PubChem search request.
- Product claims in `README.md` and `documentation/` compared with implementation paths.

Installer code signing and publisher reputation are excluded by product decision.

## Overall assessment

The implementation is technically healthy: builds, security checks, parser tests, export smoke tests, native Apple builds, and Windows packaging pass. The remaining work is concentrated in launch closure, measurable product outcomes, scientific validation breadth, and end-to-end workflow confidence.

The project is suitable for controlled internal or invited beta use. It should not be presented as a broadly validated professional workflow until the launch-blocking items below are closed or explicitly scoped out in the product copy.

## Findings

### P0 - No distributable GitHub release is published

**Intent:** The desktop updater and release documentation treat GitHub Releases as the source of truth for Windows installers.

**Implemented reality:** The workflow publishes only for a version tag, the live manifest reports `releasePublished: false`, and the repository currently has no GitHub release. The installer exists as a CI artifact and local build, but users do not have a stable public release URL.

**Impact:** A public launch has no durable installer channel, and the in-app update path cannot demonstrate a complete release-to-update loop.

**Action:** When the build is approved, create and push the matching signed Git tag, verify the generated Setup and Portable assets, deploy a manifest that marks the published build correctly, then test update detection from the previous version.

**Classification:** Launch-blocking only when public Windows distribution is intended. It is acceptable while the product remains an internal beta.

### P1 - Product telemetry cannot currently calculate the stated funnel reliably

**Intent:** The verification document defines first-run completion, calculation completion, and export completion targets.

**Implemented reality:** Every event hashes all attributes into the KV key. Because the attributes include a random session journey, most sessions create unique keys instead of useful aggregate buckets. There is also no protected read/reporting path. The KV read-then-write counter is approximate under concurrent updates.

**Impact:** The team can collect events but cannot reliably answer whether the 60%, 80%, and 90% targets are met. Launch decisions would still depend on anecdotes.

**Action:** Store event rows or journey transitions in a queryable backend with retention, and separately maintain low-cardinality daily aggregates. Add a maintainer-only report that calculates opt-in coverage, first-run completion, calculation success by engine/task, time-to-result, and export conversion.

**Classification:** Launch-blocking for a metrics-gated beta; otherwise a fast-follow required before growth decisions.

### P1 - Core UI workflows are not covered by interaction tests

**Intent:** The shipping documentation presents login, engine setup, calculation progress, cancellation, queue recovery, project persistence, and exports as release-gated user flows.

**Implemented reality:** `npm test` runs Node contract and parser scripts. Electron smoke verifies only that the app starts and exposes the desktop bridge. The sole TypeScript UI test imports Vitest, but Vitest is not installed and that test is not invoked by the test command.

**Impact:** A release can pass CI while buttons, modal transitions, scroll behavior, selected-engine persistence, account redirects, or progress updates are broken.

**Action:** Add Playwright coverage for web flows and Electron Playwright coverage for desktop-only flows. At minimum cover first launch, login callback, molecule search, engine selection, unavailable-engine recovery, calculation cancellation, queue restore, and every export menu.

**Classification:** Launch-blocking for broad distribution because these are the workflows that have accumulated the most UI changes.

### P1 - Scientific validation is narrower than the professional feature surface

**Intent:** ChemVault provides a professional bridge across Gaussian task templates, xTB, PySCF, and ORCA, with preflight checks and parsed scientific outputs.

**Implemented reality:** Input routes and parser fixtures are tested, and a guarded Gaussian water calculation exists. There is no independent benchmark matrix across representative elements, charges, multiplicities, methods, basis sets, optimization/frequency tasks, and engine revisions. Preflight accepts method and basis inputs but does not validate basis coverage for the loaded elements.

**Impact:** Unsupported method/basis combinations reach the engine and waste time, while parser or unit regressions outside the current fixtures can be missed. The engine still determines the science, but ChemVault is responsible for generating and interpreting the job correctly.

**Action:** Add an engine compatibility precheck and a versioned benchmark set with expected tolerances. Start with neutral/ionic/open-shell small molecules, common organic elements, one metal-containing case, optimization, frequency, and failed-convergence outputs.

**Classification:** Launch-blocking for a professional claim; fast-follow if the product is explicitly labeled beta and bridge-only.

### P1 - Library synchronization and permission behavior are descriptive rather than functional

**Intent:** The account area presents Library Sync, saved-project entitlement, and upload permission state from ChemVault User.

**Implemented reality:** The cloud-library feature flag is displayed, but molecule projects and calculation history remain local. The `molecule.upload` permission changes the action label to locked, yet the action still links to the same Molecule Studio page and the upload workflow is not gated by that permission.

**Impact:** Users cannot tell whether data is actually synchronized, and account entitlements appear authoritative without consistently changing behavior.

**Action:** Choose one product contract. Either implement authenticated project synchronization and actual permission enforcement, or rename the section to Local Library, remove the sync implication, and treat upload as an unrestricted local capability.

**Classification:** Fast-follow for local-first beta; launch-blocking if paid tiers or cross-device library sync are advertised.

### P2 - Engine readiness does not include a user-facing scientific self-test

**Intent:** Users should know that a configured engine is ready before committing to a long calculation.

**Implemented reality:** Engine discovery and version probes exist, and developer live-test scripts can run canonical water calculations. The application does not expose an equivalent one-click engine self-test with expected runtime and result tolerance.

**Impact:** A path/version probe can pass even when environment variables, scratch permissions, license setup, auxiliary tools, or runtime dependencies are broken.

**Action:** Add `Test engine` beside each configured engine. Run a tiny bundled water job, show live output, verify termination and an expected energy range, then save the test timestamp and engine version.

### P2 - Operational monitoring stops at CI

The repository documents that no production synthetic monitor checks PubChem, RCSB, ChemVault User, cloud quantum, and release metadata end to end. CI cannot detect an upstream outage, expired authorization contract, or broken production binding after deployment.

**Action:** Add a scheduled synthetic monitor with non-sensitive canary requests and alerts. Keep cloud quantum disabled unless its private backend, quota, and authorization checks pass.

### P2 - The quantum workspace has become a high-change single component

The main professional quantum function spans roughly two thousand lines with very high state and branch complexity. This does not prove a visual defect, but it increases the probability that hierarchy, collapsed state, progress UI, and export controls regress together.

**Action:** Split the workflow by product state: engine and readiness, calculation setup, active run, result review, and exports/history. Give each state a focused component and interaction test rather than adding more conditional blocks to the same surface.

### P3 - README scope has drifted from product-only documentation

The README again includes install commands, build commands, release directories, and packaging output. This conflicts with the established requirement that the public README describe website and app functionality without implementation or setup detail.

**Action:** Keep product capabilities in README and move developer/build instructions to a maintainer document under `documentation/`.

### P3 - Third-party distribution notices need an explicit check

The source-available license correctly reserves ChemVault rights and defers third-party components to their own terms, but there is no repository-level third-party notice inventory for the dependencies bundled in the website and Electron package.

**Action:** Generate and review a third-party notices file for shipped dependencies and include it in the desktop application or installer where required by those licenses.

## Pre-mortem

Assume the public desktop launch failed after two weeks.

### Tigers

- Users could not find a stable official installer because no GitHub release was published.
- A UI regression passed CI because startup smoke did not exercise the workflow.
- Users lost confidence after an unsupported basis/element combination failed late or a parser missed an output variant.
- The team could not explain adoption or failure rates because telemetry was not queryable as a funnel.

### Elephants

- Desktop login and OAuth depend on a live cross-service cookie/proxy contract but have no production synthetic test.
- Account library language may create an expectation of cloud synchronization that the local-first implementation does not meet.
- Engine installation succeeds on the maintainer machine but fails on managed corporate Windows devices due to proxy, antivirus, permissions, or Python policy.

### Paper tigers

- The absence of bundled commercial engines is not a defect; user-licensed Gaussian and ORCA integration is the correct boundary.
- Apple does not need to reproduce the Windows Gaussian bridge in the current product scope.
- Approximate browser electrostatics are acceptable while they remain clearly labeled and are not presented as DFT or ab initio results.

## Recommended order

1. Decide whether this is still an internal beta or a public Windows release.
2. Add UI/Electron interaction tests before further quantum-workspace expansion.
3. Build the scientific benchmark and method/basis compatibility matrix.
4. Replace the current telemetry storage shape with a queryable funnel and reporting view.
5. Resolve the local-library versus cloud-sync product contract.
6. Add the in-app engine self-test and production synthetic monitoring.
7. Clean up README scope and third-party notices.
